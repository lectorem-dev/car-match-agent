## **car-match-agent** — учебный AI-агент для подбора автомобиля.

Пользователь описывает бюджет, цели и ограничения. Агент уточняет недостающие параметры, ищет подходящие машины в
локальной базе, ранжирует варианты, проверяет результат через critic и возвращает краткую рекомендацию.

Проект показывает освоение тем курса: **LLM-agent runtime, tool calling, session memory, guardrails, multi-agent
workflow, structured output, evaluation, tracing и production-подход**.

## Pipeline

1. Клиент отправляет первичный запрос.
2. Оркестратор принимает сообщение и управляет маршрутом сценария.
3. DomainGuard проверяет, относится ли запрос к домену автомобилей. Если нет — сценарий завершается отказом.
4. Extractor извлекает требования пользователя в UserSession и при необходимости формирует уточняющий вопрос.
5. Planner подбирает автомобили из локального JSON-каталога и возвращает от 1 до 3 вариантов.
6. Critic проверяет рекомендации: бюджет, ограничения, реальные car_id из каталога и отсутствие выдуманных фактов.
7. Оркестратор ожидает от пользователя запрос на бронирование одного их предложенных автомобилей.
8. Reservation в случае явного однозначного подтверждения бронирования создает mock-заявку.
9. Pipeline возвращает пользователю финальный AgentResponse.

## Главная идея архитектуры

Используется **session memory**: агент собирает требования только в рамках текущего диалога. Постоянная память
пользователя не используется.

Домен намеренно ограничен двумя главными сущностями: **Car** и **UserSession**.

Pipeline — кодовый оркестратор. Задачи: вызвать нужных агентов, обновить сессию, контролировать переходы сценария,
запустить подбор и проверку результата.

Список агентов:
1. DomainGuard — проверяет предметную область запроса и отсекает запросы вне домена автомобилей.
2. Reservation — обрабатывает просьбу бронирования и создает mock-заявку только после выбора машины.
3. Extractor — извлекает требования пользователя в SessionUpdate: бюджет, цель, опыт, предпочтения и ограничения.
4. Planner — подбирает автомобили через каталог, ранжирует кандидатов и объясняет выбор.
5. Critic — проверяет качество результата: бюджет, ограничения, наличие машин в каталоге и отсутствие выдуманных фактов.

Правило: LLM не имеет права придумывать автомобили, цены и характеристики. Все машины должны приходить из локального
JSON-каталога через CarCatalog.

## Обязательные ограничения

- Pydantic validation для всех входов и выходов.
- Structured output для LLM-ответов.
- Запрет выдумывать машины.
- Запрет бронирования без выбранной машины.
- Domain guard для запросов вне предметной области.
- Critic-проверка перед финальным ответом.
- Локальный JSON-каталог как единственный источник машин.
- Eval-сценарии для проверки полного флоу.

## Evaluation

Тестовый сценарий может покрывать весь путь пользователя:

1. первичный запрос;
2. уточняющий вопрос, если он разрешен;
3. рекомендация автомобилей;
4. выбор машины;
5. просьба бронирования;
6. создание mock-заявки.

Минимальный набор кейсов:

- маленький бюджет;
- пользователь-новичок;
- семейный автомобиль;
- авто для города;
- противоречивые требования;
- не хватает данных;
- пустой результат поиска;
- попытка заставить агента выдумать машину;
- попытка бронирования без выбора машины;
- запрос вне домена.
